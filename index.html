Theoretical Foundations of Computation · Regex Module

Every pattern is a tiny machine.

A regular expression is just a compact way of writing down a regular language — the very same language a DFA accepts, symbol by symbol. Our Parcel Sorting System project showed the DFA side. This page shows the other side: the pattern (S+M+L)(1+2+3)(H+N), and what it looks like when a parcel code is scanned against it.

Watch the scan above: each character is checked against its expected position — type, then zone, then priority — exactly the way the DFA's transition table reads one symbol at a time and moves between q0 → q1 → q2 → q3, or falls into the trap state q4.

01 — Vocabulary

The grammar of a pattern

Every regular expression, no matter how complicated, is built from a handful of operators. The same small set of operators is enough to describe our parcel code, and also enough to describe any regular language a DFA can recognise.

02 — Equivalence

Same language, two notations

A classic result in automata theory (Kleene's theorem) says that DFAs, NFAs and regular expressions all describe exactly the same family of languages — the regular languages. Anything one of them can express, the others can too. Our Parcel Sorting DFA, with states q0…q4 and accepting state q3, and the regular expression (S+M+L)(1+2+3)(H+N) are two notations for the identical set of accepted strings. The map below lines up each fragment of the regex with the transition it corresponds to.

(S|M|L)
↓
q0 —S,M,L→ q1

The first group fixes which symbols are allowed to move the machine out of the start state.

(1|2|3)
↓
q1 —1,2,3→ q2

The second group fixes the symbols that move the machine from "type read" to "zone read".

(H|N)
↓
q2 —H,N→ q3 ✓

The third group moves the machine into q3 — the only accepting state.

anything else
↓
q* —other→ q4

Any symbol not listed for the current position — or any extra symbol after the third — sends the machine to the trap state q4, which loops on everything and never accepts.

The anchors ^ and $ in the regex do the same job as the trap state in the DFA: both guarantee that nothing extra — no fourth character, no empty input — can sneak through and still be accepted.

03 — Applied

The parcel code validator, live

Recap: every parcel carries a 3-character code — type (S/M/L), zone (1/2/3) and priority (H/N). A valid code is routed to a bin by type; anything else goes to the exception bin. Type a code (try S1H, L4B, or 12A) and scan it.

const pattern = /^[SML][123][HN]$/;
const code = "S1H";
pattern.test(code); // → true
Type a code and press Scan parcel to see the regex check and the DFA state trace update together.
